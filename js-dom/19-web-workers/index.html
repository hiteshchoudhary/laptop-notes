Example 4 — What Can Be Sent to Workers

postMessage uses Structured Clone, NOT JSON. Supports more types but still has restrictions.

CAN be sent

  • Strings, Numbers, Booleans
  • null, undefined
  • Date objects (preserved)
  • RegExp (preserved)
  • Map, Set (preserved)
  • ArrayBuffer, Blob
  • Nested objects/arrays

CANNOT be sent

  • Functions / closures
  • DOM nodes (document.body)
  • Symbols
  • Class instances lose prototype

Transferable (zero-copy)

  • ArrayBuffer (MOVED, not copied)
  • MessagePort
  • OffscreenCanvas
  • Sender loses access!

Example 5 — Worker Limitations & Available APIs

Workers are sandboxed. No DOM prevents race conditions. But many APIs ARE available.

CANNOT Access

  • window, document
  • document.cookie
  • localStorage, sessionStorage
  • alert(), confirm(), prompt()
  • Parent page variables

CAN Access

  • self (worker global scope)
  • fetch() — HTTP requests
  • IndexedDB — database
  • setTimeout / setInterval
  • WebSocket — real-time
  • crypto — encryption
  • console — debugging
  • navigator.onLine
  • Blob, FileReader

Key Takeaways

  1. Workers run JS in a SEPARATE THREAD, preventing heavy computation from blocking the UI.
  2. Communication via postMessage/onmessage. Data is COPIED using Structured Clone.
  3. Transferable objects (ArrayBuffer) can be MOVED — zero copy but sender loses access.
  4. Workers CANNOT access: DOM, window, localStorage, cookies. CAN access: fetch, IndexedDB, WebSocket, crypto, timers.
  5. Inline workers use Blob + URL.createObjectURL() for small utility workers.
  6. SharedWorkers are shared across tabs of the same origin.
  7. Always terminate workers when done — they consume real system resources.
  8. Use navigator.hardwareConcurrency to size worker pools.
  9. Use request IDs (RPC pattern) for concurrent requests.
  10. Ideal for: CSV export, encryption, image processing, data sorting, mathematical computation.
